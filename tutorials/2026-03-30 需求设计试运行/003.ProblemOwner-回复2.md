让 Agent 在 GitLab 里工作。
智能体执行任务是根据issue的任务负责人和任务状态来决定的，当任务进入到NEW、并且分配给了问题定义者，则会触发智能体执行任务。智能体会根据issue的描述和相关信息来执行相应的操作。
智能体之间存在依赖关系与执行顺序，在需求设计阶段的依赖顺序是 ProblemAgent -> ValidatorAgent -> ProblemOwnerAgent -> ProductAgent -> ROIAgent
一个 Issue 不仅对应一个智能体，他会有多个不同角色基于他的背景配置发出各自立场的评价。
任务是由事件驱动，例如当issue分配到指定智能体时。
Issue 是任务单元，还可以继续分拆为更小的sub issue。
状态机是统一的，
- `待整理`：需求输入已记录，尚未收敛
- `讨论中`：正在形成统一方案
- `待裁决`：已形成计划，等待董事会裁决
- `已通过`：可进入研发执行链
- `待拆解`：已进入研发仓，待拆为可执行任务
- `待开发`：任务已就绪，尚未开工
- `开发中`：已有代理开始处理
- `代码审查中`：工作已提交，等待 MR 审查
- `验收中`：代码审查通过，等待产品或测试验收
- `已阻塞`：因依赖、权限、资源或决策问题被阻塞
- `已完成`：工作已完成并验收
- `已取消`：工作被明确终止

问题标题

多智能体协作过程中存在重复建设与流程割裂问题

⸻

背景
	•	当前存在使用 AI 构建产品与系统的需求
	•	AI 降低了开发门槛，使基础能力实现成本下降
	•	团队在推进产品过程中存在自建基础设施的行为
	•	GitLab 已作为代码与任务管理平台存在于研发流程中
	•	GitLab 支持 Issue 作为任务记录与协作载体
	•	多智能体协作被作为一种工作模式引入
	•	存在对多智能体统一管理与协作机制的需求
	•	存在对协作过程进行记录与追踪的需求
	•	存在对智能体进行统一配置管理的需求（Skill、参数、规则等）

⸻

核心问题

当前在多智能体协作系统构建过程中，存在从零构建基础能力的倾向，导致资源消耗集中在非核心价值部分，同时未将 GitLab 统一作为多智能体协作的执行环境与运行载体，缺乏明确的任务触发机制、执行顺序与状态流转规则

⸻

目标
	•	多智能体直接在 GitLab Issue 中执行任务
	•	每个智能体的执行基于以下条件触发：
	•	Issue 被分配给该智能体
	•	Issue 状态进入指定阶段（如：待整理 / NEW）
	•	所有智能体输入与输出统一记录在 Issue 中，且：
	•	所有记录具备时间戳
	•	所有记录具备执行主体标识
	•	多智能体协作过程具备可还原性：
	•	可基于 Issue 记录完整还原执行流程
	•	智能体执行具备明确顺序与依赖关系：
	•	ProblemAgent → ValidatorAgent → ProblemOwnerAgent → ProductAgent → ROIAgent
	•	每个任务具备统一状态机，状态可被唯一判断

⸻

范围

包含：
	•	基于 Issue 的任务触发机制（分配 + 状态驱动）
	•	多智能体在同一 Issue 内的协作规则
	•	智能体执行顺序与依赖关系定义
	•	Issue 作为任务单元及其可拆分机制（Sub Issue）
	•	统一状态机定义与状态流转规则
	•	智能体配置的集中管理（Skill、参数、规则）

不包含：
	•	状态展示方式（UI / 交互）
	•	底层 AI 模型能力开发
	•	GitLab 本身功能改造
	•	具体业务流程定义

⸻

非范围
	•	不涉及自研代码托管系统
	•	不涉及替代 GitLab 的任务管理能力
	•	不涉及具体智能体能力实现细节
	•	不涉及 UI 设计与交互实现

⸻

约束条件
	•	必须基于 GitLab Issue 作为唯一任务与数据载体
	•	智能体必须以独立账号参与协作
	•	不允许改造 GitLab 核心功能
	•	所有执行过程必须记录在 Issue 中
	•	状态流转必须符合统一状态机
	•	任务触发必须基于事件（状态变更 / 分配）

⸻

关键风险/歧义

（已降级为非阻塞风险）
	•	多智能体在同一 Issue 中的并发行为是否会产生冲突
	•	Issue 粒度在复杂任务中是否足够（是否需要强制拆分规则）
	•	状态机在不同业务场景下是否需要扩展
	•	智能体权限与 GitLab 权限体系的映射细节未定义
	•	控制系统是否具备强制干预能力未明确

⸻

验收口径
	•	输出包含全部字段且结构完整
	•	每个字段内容可被独立理解
	•	所有约束条件为客观限制
	•	所有风险与不确定点已列出
	•	无方案设计与实现描述

	•	任一时刻可通过 Issue 判断：

	•	上一步执行记录
