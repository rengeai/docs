⸻

1. 方案标题

基于 GitLab Issue 的多智能体协作执行系统

⸻

2. 目标用户
	•	使用 GitLab 进行研发协作的产品/研发团队
	•	需要引入多智能体参与工作流的团队负责人 / AI系统设计者

⸻

3. 用户场景
	•	团队在 GitLab 中创建一个 Issue 作为任务起点
	•	将 Issue 分配给 ProblemAgent，并设置状态为 NEW
	•	多个智能体按照既定顺序，在同一个 Issue 中依次执行任务
	•	每个智能体在 Issue 中输出结果并更新状态
	•	最终完成从问题定义 → 校验 → 收敛 → 产品方案 → ROI 的完整链路
	•	任意时刻团队成员可以通过 Issue 查看当前进度与历史记录

⸻

4. 设计目标

（完全承接问题定义，不新增目标）
	•	多智能体直接在 GitLab Issue 中执行任务
	•	基于「分配 + 状态」触发智能体执行
	•	所有输入输出统一记录在 Issue 中（带时间戳 + 执行主体）
	•	协作过程可被完整还原
	•	执行顺序与依赖关系明确
	•	任务状态具备唯一可判断性

⸻

5. 核心方案

构建一个以 GitLab Issue 为核心运行载体的多智能体执行框架：
	1.	Issue 即任务单元
	•	每个 Issue = 一个完整协作链
	•	所有行为都发生在 Issue 内
	2.	事件驱动执行机制
	•	执行触发条件：
	•	Issue 被分配给某个 Agent
	•	Issue 状态进入该 Agent 对应阶段
	3.	顺序化 Agent 流转
	•	固定执行链：
ProblemAgent → ValidatorAgent → ProblemOwnerAgent → ProductAgent → ROIAgent
	•	每一步完成后：
	•	写入结果
	•	更新状态
	•	转交下一个 Agent
	4.	结构化记录协议
	•	所有输出必须：
	•	带 Agent 标识
	•	带时间戳
	•	使用固定结构模板
	5.	统一状态机
	•	每个 Issue 必须处于明确状态
	•	状态驱动执行，而不是人为推动
	6.	集中配置管理
	•	每个 Agent 的：
	•	Skill
	•	Prompt
	•	规则
	•	统一管理，不分散在 Issue 中

⸻

6. 功能列表
	1.	Issue 任务触发机制
	•	基于分配 + 状态触发执行
	2.	Agent 执行调度规则
	•	定义执行顺序与依赖关系
	3.	状态机管理
	•	定义状态集合与流转规则
	4.	结构化输出规范
	•	强制 Agent 使用标准模板输出
	5.	执行记录系统
	•	在 Issue 中记录所有执行历史
	6.	Agent 配置管理
	•	集中维护 Agent 能力与规则
	7.	Issue 拆分机制（Sub Issue）
	•	支持复杂任务拆分

⸻

7. MVP范围

第一阶段只做「能跑通完整链路」：
	•	单 Issue 完整执行链（Problem → ROI）
	•	固定顺序 Agent 执行（不支持动态编排）
	•	基础状态机（线性流转，不支持复杂分支）
	•	Issue 内结构化记录
	•	基于「分配 + 状态」的触发机制

👉 本质目标：
先证明“能跑通”，不是“跑得优雅”

⸻

8. 非MVP范围

明确不做（否则你会失控）：
	•	并发执行控制机制
	•	状态回退机制
	•	动态 Agent 编排
	•	UI 展示优化
	•	权限系统深度集成
	•	自动拆分 Issue 的智能策略
	•	多 Issue 协同编排

⸻

9. 用户流程
	1.	创建 Issue
	2.	设置状态为 NEW
	3.	分配给 ProblemAgent
	4.	ProblemAgent 执行：
	•	输出问题定义
	•	更新状态 → VALIDATING
	•	分配给 ValidatorAgent
	5.	ValidatorAgent 执行：
	•	输出校验结果
	•	更新状态 → CONVERGING
	•	分配给 ProblemOwnerAgent
	6.	ProblemOwnerAgent 执行：
	•	输出最终问题定义
	•	更新状态 → DESIGNING
	•	分配给 ProductAgent
	7.	ProductAgent 执行：
	•	输出产品方案
	•	更新状态 → ROI
	•	分配给 ROIAgent
	8.	ROIAgent 执行：
	•	输出价值评估
	•	更新状态 → DONE

⸻

10. 交互原则
	•	状态先于执行（没有状态，不执行）
	•	一次只允许一个 Agent 执行（避免冲突）
	•	输出必须结构化（否则不可用）
	•	执行必须可追溯（所有行为有记录）
	•	流程必须收敛（禁止无限循环）

⸻

11. 依赖与约束
	•	必须依赖 GitLab Issue 作为唯一载体
	•	必须使用 GitLab 原生能力（分配 / 状态）
	•	Agent 必须以独立账号存在
	•	所有记录必须写入 Issue
	•	不允许绕过状态机执行

⸻

12. 风险与取舍

为什么不做并发？

👉 因为你现在连“顺序执行”都还没稳定
并发只会让系统直接失控

⸻

为什么不做状态回退？

👉 回退 = 引入复杂性 + 冲突
MVP阶段优先保证“单向确定性”

⸻

为什么不做动态编排？

👉 动态 = 不可控
你现在需要的是“确定路径”

⸻

为什么不做自动拆分？

👉 拆分策略本身就是一个复杂问题
现在人为拆分更可控

⸻

13. 验收口径

满足以下条件即方案成立：
	•	一个 Issue 可完整跑通 Agent 链路
	•	任意时刻可判断：
	•	当前状态
	•	当前 Agent
	•	上一步输出
	•	所有输出均：
	•	有结构
	•	有主体
	•	有时间
	•	无人工介入也能完成流程推进
