# 运行示例：研发需求交付

这篇文档只描述一个研发需求如何从 `docs` 仓库走到研发总仓的里程碑和 MR，不混入线上事故处理或产品方案讨论。

## 场景设定

公司名称为 `NoteFlow AI`，目标是：

- 打造月经常性收入达到 100 万美元的排名第一的 AI 笔记应用

当前项目为：

- `consumer/noteflow-app`（研发总仓，包含多个 submodule）

当前公司级需求仓库为：

- `noteflow/docs`

当前里程碑为：

- `2026 年第二季度转化率提升`

## 参与代理

- `产品经理代理`：负责创建议题、维护需求优先级和里程碑
- `研发经理代理`：负责拆解任务、分配执行代理、审核合并请求
- `前端代理`：负责前端实现
- `后端代理`：负责后端实现
- `测试代理`：负责验证和回归

## 研发流转示例

下面用“新增 AI 摘要功能”来展示从需求输入到研发交付的完整链路。

### 第一步：在 `docs` 仓库创建需求 Issue

`产品经理代理` 在 `noteflow/docs` 创建一个类型为“议题”的 Issue：

- 标题：为笔记详情页增加 AI 自动摘要功能
- 描述：帮助用户快速理解长笔记内容，提升留存与付费转化
- 当前阶段：`待整理`
- 目标项目：`consumer/noteflow-app`
- 决策者：`董事会`
- 计划版本：`plan-noteflow-summary-v1`

这个 Issue 是需求载体，负责定义要做什么以及为什么做。

### 第二步：形成需求计划 MR 并进入裁决

多个员工在该 Issue 下讨论后，由 `决策者员工` 形成计划文档并发起 MR，至少包含：

- 背景
- 问题
- 范围
- 不做什么
- 验收标准
- 拆解建议

随后 `董事会` 执行强门禁裁决。只有裁决通过，计划 MR 才能合并。

### 第三步：人工确认后同步到研发总仓

`项目经理代理` 在人工确认后，将该计划同步到 `consumer/noteflow-app`，并创建：

- 里程碑：`2026 年第二季度转化率提升`
- 父 Issue：AI 摘要功能交付
- 子任务：交互设计、接口实现、前端展示、测试验收

这些对象统一记录三元追溯字段：

- `docs_issue_id`
- `plan_doc_path`
- `plan_commit_sha`

### 第四步：研发代理原子认领并开发

当“任务 2：实现摘要生成接口”进入 `待开发` 后，`后端代理` 在心跳中尝试认领任务，将状态推进到 `开发中`。

如果另一个代理同时认领同一个任务，系统应返回 `409 Conflict`。

### 第五步：通过 MR 完成审查与验收

`后端代理` 完成开发后发起 MR，必须回链父 Issue 并继承三元追溯字段。随后：

- `代码审查代理` 推进 `代码审查中`
- `测试代理` 与 `产品经理代理` 推进 `验收中`
- 验收通过后任务进入 `已完成`

## 这个示例的重点

- `docs` 仓库负责需求讨论、计划成文与裁决
- 研发总仓负责里程碑、任务和代码交付
- 董事会裁决是强门禁
- 项目经理代理采用人工确认后创建执行对象
- 三元追溯字段保证计划与代码可反查
