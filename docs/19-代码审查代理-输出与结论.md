# 代码审查代理：输出与结论

## 输出内容

代码审查结果应按严重程度组织。每个问题至少包含：

- 严重级别
- 问题标题
- 文件位置
- 问题说明
- 修复建议

推荐格式如下：

```text
[严重] 源码中存在硬编码 API Key
文件: src/api/client.ts:42
问题: API Key 被直接写入源码，提交后会永久进入 Git 历史。
建议: 改为环境变量读取，并补充到 .env.example 或相关配置中。
```

## 审查总结

每次审查结束后，都应输出统一总结：

```text
## 审查总结

| 严重级别 | 数量 | 状态 |
|----------|------|------|
| 严重     | 0    | 通过 |
| 高       | 2    | 警告 |
| 中       | 3    | 提示 |
| 低       | 1    | 备注 |
```

## 结论标准

- 通过：没有严重或高优先级问题
- 警告：存在高优先级问题，但不包含严重问题
- 阻断：存在严重问题，必须修复后才能合并

## 项目规则对齐

如果仓库中存在项目约定文件，例如 `CLAUDE.md` 或其他规则文件，代码审查代理还应额外检查：

- 文件大小上限
- 表情符号使用约束
- 不可变写法约束
- 数据库与迁移规范
- 错误处理模式
- 状态管理约定
