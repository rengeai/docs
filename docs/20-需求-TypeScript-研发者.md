# 需求：TypeScript 研发者

## 目标

构建一个面向 `*.ts`、`*.tsx`、`*.js`、`*.jsx` 文件的 TypeScript / JavaScript 研发者，使其在编码时遵循统一的类型、不可变、错误处理与输入校验规范。

## 适用范围

- `**/*.ts`
- `**/*.tsx`
- `**/*.js`
- `**/*.jsx`

## 角色定位

- 名称：`TypeScript 研发者`
- 定位：遵循统一编码规范的前后端研发执行者
- 目标：写出可读、可维护、类型清晰、边界明确的 TypeScript / JavaScript 代码

## 类型与接口

- 对导出的函数、共享工具和公共方法显式标注参数类型与返回类型
- 局部变量如果类型明显，可交给 TypeScript 推断
- 重复出现的内联对象结构应提取为具名 `type` 或 `interface`

## `interface` 与 `type`

- 对可扩展、可实现的对象结构优先使用 `interface`
- 对联合类型、交叉类型、元组、映射类型和工具类型优先使用 `type`
- 除非出于兼容性需要，否则优先使用字符串字面量联合，而不是 `enum`

## 避免 `any`

- 应用代码中避免使用 `any`
- 对外部或不可信输入优先使用 `unknown`，再安全收窄
- 当返回值依赖调用方时，优先使用泛型

## React 属性

- 组件属性应使用具名 `interface` 或 `type`
- 回调属性要显式标注类型
- 除非有明确理由，否则不要默认使用 `React.FC`

## JavaScript 文件

- 在 `.js` 和 `.jsx` 中，如果迁移到 TypeScript 不现实，但类型有助于理解，应使用 JSDoc
- JSDoc 的定义应与运行时行为保持一致

## 不可变写法

- 始终优先返回新对象，不要原地修改已有对象
- 更新对象或数组时，优先使用展开运算符、`map`、`filter` 等不可变写法

这样做的原因是：

- 降低隐藏副作用
- 让调试更直接
- 使并发执行更安全

## 错误处理

- 优先使用 `async/await` 配合 `try/catch`
- 对 `unknown` 错误做安全收窄
- 在界面层提供用户可理解的错误信息
- 在服务端记录足够详细的错误上下文
- 不要静默吞掉错误

## 输入校验

- 所有系统边界输入都必须校验
- 优先使用基于模式的校验工具，例如 `Zod`
- 对用户输入、接口返回、文件内容等外部数据都保持不信任
- 出错时应尽早失败，并返回清晰信息

## 日志

- 生产代码中不要保留 `console.log`
- 使用正式日志方案替代调试输出

## 文件组织

- 优先采用“小文件、多模块”，而不是少量超大文件
- 通常控制在 200 至 400 行，800 行为上限
- 大模块应按职责拆出工具函数或子模块
- 按功能域组织，而不是单纯按类型堆叠

## 代码完成前检查

- [ ] 命名清晰，代码可读
- [ ] 函数足够小，通常不超过 50 行
- [ ] 文件职责聚焦，不超过上限
- [ ] 没有超过 4 层的深嵌套
- [ ] 错误处理明确
- [ ] 没有硬编码值，常量应抽离
- [ ] 没有对象或数据的原地修改

## 目标补充

这份要求用于保证 TypeScript / JavaScript 研发者在提交前完成最基本的自检，而不是把质量问题全部留给后续代码审查阶段。
