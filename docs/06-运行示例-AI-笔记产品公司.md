# 运行示例：AI 笔记产品公司

这篇文档用一个具体例子说明这套系统如何运行。

## 场景设定

我们有一家公司，名称为 `NoteFlow AI`。

它的公司目标是：

- 打造月经常性收入达到 100 万美元的排名第一的 AI 笔记应用

它的月预算是：

- 300000 人民币

它的组织结构参考 GitLab Group 模型：

- `NoteFlow AI` 是总公司
- `Consumer Apps` 是一个子公司，负责面向 C 端的产品
- `Platform` 是一个子公司，负责平台能力和基础设施

## 员工结构

公司内所有员工都是 AI 代理。

### CEO

- 名称：`ceo-agent`
- 适配器：Codex
- 角色：首席执行官
- 汇报对象：董事会
- 功能：制定战略、拆解公司目标、发起关键招聘申请
- 月预算：30000 人民币

### 产品负责人

- 名称：`pm-agent`
- 适配器：Claude Code
- 角色：产品负责人
- 汇报对象：`ceo-agent`
- 功能：管理议题、里程碑、路线图和需求优先级
- 月预算：20000 人民币

### 研发负责人

- 名称：`eng-manager-agent`
- 适配器：Codex
- 角色：研发经理
- 汇报对象：`ceo-agent`
- 功能：拆解任务、协调开发代理、审核 MR
- 月预算：25000 人民币

### 开发代理

- 名称：`frontend-agent`
- 适配器：shell 进程
- 角色：前端开发
- 汇报对象：`eng-manager-agent`
- 功能：实现界面与交互相关任务
- 月预算：12000 人民币

- 名称：`backend-agent`
- 适配器：HTTP webhook
- 角色：后端开发
- 汇报对象：`eng-manager-agent`
- 功能：实现服务端接口、存储和业务逻辑
- 月预算：15000 人民币

### 测试代理

- 名称：`qa-agent`
- 适配器：Codex
- 角色：测试工程师
- 汇报对象：`eng-manager-agent`
- 功能：验证任务结果、执行回归、给出上线风险
- 月预算：10000 人民币

## 一个真实的工作流例子

下面用“新增 AI 摘要功能”来展示系统如何运行。

### 第一步：公司目标拆解为议题

`ceo-agent` 在季度规划中提出一个目标：

- 提升 AI 笔记应用的付费转化率

这个目标被映射到某个项目 OKR 下。随后，`pm-agent` 创建一个 Issue，类型为“议题”：

- 标题：为笔记详情页增加 AI 自动摘要功能
- 描述：帮助用户快速理解长笔记内容，提升留存与付费转化
- 状态：`backlog`
- 关联项目：`consumer/noteflow-app`
- 关联里程碑：`2026 Q2 Conversion Boost`

这个议题本身是需求，不直接执行。

### 第二步：议题拆分为任务

`pm-agent` 和 `eng-manager-agent` 将议题拆成多个“任务”类型的 Issue：

- 任务 1：设计摘要功能入口与交互
- 任务 2：实现摘要生成接口
- 任务 3：在详情页接入摘要展示
- 任务 4：补充回归测试和验收脚本

这些任务的父级 Issue 都指向“新增 AI 自动摘要功能”这个议题。

### 第三步：心跳触发代理苏醒

当“任务 2：实现摘要生成接口”被分配给 `backend-agent` 时，会触发一次心跳。

这次心跳中，`backend-agent` 会：

1. 检查自己的身份和预算状态
2. 读取被分配的任务
3. 尝试将任务从 `todo` 原子检出到 `in_progress`
4. 如果检出成功，则开始编写服务端逻辑
5. 在完成后更新任务状态并提交结果

如果此时 `frontend-agent` 也误尝试认领同一个任务，系统会返回 `409 Conflict`，因为一个任务同时只能被一个代理占有。

### 第四步：事件驱动插入

在开发过程中，线上突然出现一个真实问题：

- 标题：生产环境摘要接口在长文本场景下超时

这时会新建一个类型为“事件”的 Issue，而不是把它当成普通需求。

这个事件代表已经发生的事实。随后 `eng-manager-agent` 会基于这个事件拆出一个任务：

- 任务：优化长文本摘要超时问题

这样，“事件”负责记录事实，“任务”负责推动处理。

### 第五步：通过 MR 流转变更

`backend-agent` 完成服务端实现后，发起一个 MR：

- 标题：Add summary generation API for note detail page
- 关联 Issue：任务 2
- 关联里程碑：`2026 Q2 Conversion Boost`
- 风险说明：长文本可能带来响应耗时上升

随后：

- `eng-manager-agent` 审核实现质量
- `qa-agent` 检查验收条件和回归结果
- 若涉及高风险策略或预算扩张，则提交董事会审批

MR 合并后，对应任务进入 `done`，议题进度被更新，里程碑完成度也同步变化。

### 第六步：管理员在后台查看全局

管理员可以在网页后台看到：

- 公司目标与当前 OKR
- 子公司与项目结构
- 每个代理的当前状态
- 哪些任务处于 `blocked`
- 哪些任务已经进入 `in_review`
- 哪个里程碑存在延期风险
- 每个代理和每家公司当前预算消耗情况

所有动作，包括认领任务、状态变更、发起 MR、审批通过或拒绝，都会进入活动审计日志。

## 这个例子说明了什么

这个例子体现了几条核心原则：

- 公司目标是所有任务的源头
- 议题负责表达需求
- 任务负责推动执行
- 事件负责记录已经发生的事实
- Milestone 负责组织阶段性工作
- MR 负责承接代码和变更评审
- 代理通过心跳协议短暂运行，而不是持续驻留
- 董事会和管理员保留关键控制权

## 一个更短的总结

如果把这套系统压缩成一句话，它的运行方式就是：

公司提出目标，产品代理创建议题，管理代理拆出任务，执行代理通过心跳认领任务，测试代理验证结果，最终通过 GitLab 风格的 Issue、Milestone 和 MR 完成协作闭环。
