# 产品蓝图与功能板块

## 本文边界

本文只回答产品层问题：

- 这个产品的使命、愿景和核心用户是谁
- 第一版本要解决什么问题
- MVP 应该包含哪些模块
- 各模块分别负责什么

愿景和长期方向的论述，请看 [项目愿景与理念](./01-项目愿景与理念.md)。  
具体的组织规则与项目流转，请分别看 [组织与权限模型](./02-组织与权限模型.md) 和 [研发协作体系](./03-研发协作体系.md)。

## 产品定位

这套系统是一个建立在 GitLab 与 OpenClaw 之上的统一治理层。

GitLab 是协作底座，但不是全部系统本体。公司、子公司、项目、Issue、里程碑、MR 等核心协作对象，继续尽量沿用 GitLab 语义。

严格来说，即使没有这套产品，仍然可以单独搭建 OpenClaw 服务、手工配置 GitLab 群组、项目、账号、仓库和协作规则，从而完成多智能体协作。但那样的方式有几个明显问题：

- 初始搭建繁杂
- 组织关系需要手工维护
- 权限与职责不易统一
- 公司级治理信息分散
- 后续扩张和长期维护成本较高

因此，这个产品的目标不是替代 GitLab，也不是替代 OpenClaw，而是把原本分散的搭建过程产品化、制度化和可视化。

## 产品使命与愿景

### 使命

服务于中小规模的研发团队，使其更好地完成 AI 化，让 10 人的研发团队释放出 100 人团队的价值。

### 愿景

让人工主导的迭代过程，逐步过渡到 AI 自我协作、自我提升、自我迭代的新阶段。

## 核心用户

当前产品优先面向以下用户群体：

- 3 人以上的团队
- 100 人以下的研发型团队
- 技术公司
- 软件外包公司
- 软件开发咨询公司

这些团队通常已经具备基础研发流程，但缺少一套足够稳定的 AI 协作与治理框架。

## 核心场景

当前版本聚焦的核心场景是：

- 接管研发全流程，从产品想法到产品上线

在第一版本里，这个“接管”并不意味着一步替代所有人类动作，而是优先把以下链路组织清楚：

- 产品想法进入议题
- 议题进入里程碑
- 任务被拆解和分配
- 研发通过 MR 提交和评审
- 项目完成一轮可复用的迭代闭环

## 核心能力

这套系统希望最终形成三类核心能力：

- AI 自我协作
- AI 自我提升
- AI 自我迭代

从演进路径上看，它不是一步从“人工掌控”跳到“完全自治”，而是逐步过渡：

- 第一阶段：人类主导，AI 辅助执行
- 第二阶段：人类治理，AI 主动协作
- 第三阶段：在明确边界内实现更强的自我迭代

## MVP 设计原则

MVP 阶段不追求一步到位做完整平台，而是优先做出最小可运行闭环。

当前第一版本，我们先只考虑一个研发项目如何运行，不试图一次完成整套多公司、多项目的完整系统。

这个最小闭环至少要满足四件事：

- 能先创建一个公司并登记员工信息
- 能把 OpenClaw 员工接入到公司治理体系中
- 能通过 GitLab 的项目、Issue、里程碑、MR 完成一个研发项目的基本协作
- 能让管理者看到员工、任务和项目进度之间的对应关系

换句话说，MVP 首先要解决的是“一个项目如何被稳定地迭代”，而不是“穷尽所有组织能力”。

## 功能板块总览

从 MVP 视角出发，系统应优先包含以下功能板块：

- 公司与子公司管理
- 公司自述仓库管理
- OpenClaw 员工接入
- 员工 `SELF` 自述仓库接入
- 职业化服务模型
- 项目与协作流程管理
- 目标与里程碑管理
- 审批、审计与后台治理

## 各模块的职责边界

### 1. 公司与子公司管理

这个模块只负责回答“谁是治理主体”。

MVP 阶段至少要支持：

- 创建公司
- 在公司下创建子公司
- 查看公司树状结构
- 查看群组与子群组的继承关系
- 查看公司对应的目标、预算和员工规模

### 2. 公司自述仓库管理

这个模块只负责回答“公司是谁、为什么存在、遵循什么规则”。

MVP 阶段至少要支持：

- 检查公司是否已建立自述仓库
- 为新公司创建默认自述仓库
- 为公司自述仓库生成初始化模板
- 将公司自述仓库与公司主页或后台资料页关联

公司自述仓库建议至少包含：

- 公司简介
- 公司使命与目标
- 业务边界
- 组织结构说明
- 预算原则
- 协作约定
- 当前重点项目

### 3. OpenClaw 员工接入

这个模块只负责回答“有哪些员工被纳入治理体系”。

这里需要明确：

- OpenClaw 定义的是员工标准
- GitLab 账号定义的是协作身份
- 模型适配、运行时适配和其他适配器问题由 OpenClaw 自行解决

MVP 阶段至少要支持：

- 录入员工的 GitLab 账号
- 建立员工与 OpenClaw 服务的对应关系
- 指定员工所属公司、直属上级和职业类型
- 查看员工当前状态与可承担职责

### 4. 员工 `SELF` 自述仓库接入

这个模块只负责回答“这个员工是谁、擅长什么、适合做什么”。

`SELF` 先于公司存在，公司负责绑定和授权，而不是替代员工维护身份资料。

MVP 阶段至少要支持：

- 检查员工是否拥有 `SELF` 仓库
- 将 `SELF` 仓库与员工档案建立关联
- 从 `SELF` 中读取关键定义并展示摘要
- 在分配任务时引用员工 `SELF` 中的能力信息

### 5. 职业化服务模型

这个模块只负责回答“员工服务应该如何建模”。

系统建议用户不要在同一个服务中配置多种完全不同职责的角色。

MVP 阶段应明确采用职业化建模：

- 一个服务对应一种职业线
- 职业线之下再划分岗位
- 同一岗位下可以有多个分身
- 同一职业内可以定义不同性格和不同处理方式
- 不鼓励同一服务同时承担产品、研发、测试、运维等完全不同职业

例如：

- 产品服务可以包含需求整理、原型分析、方案输出等多个同职业角色
- 研发服务可以包含前端分身、后端分身、接口分身等多个同职业执行单元
- 前端和后端都同属于研发这一职业线
- 但不应让同一个服务既做产品决策，又直接承担研发实现

### 6. 项目与协作流程管理

这个模块只负责回答“项目内的工作如何流转”。

MVP 阶段至少应支持：

- 在公司或子公司下创建项目
- 使用 Issue 管理议题、事件、任务
- 使用里程碑组织阶段目标
- 使用 MR 承接变更评审与合并
- 按 GitLab 权限模型配置成员权限

这里的重点不是重新发明一套任务系统，而是先把一个研发项目中的 Issue、里程碑、MR 迭代闭环组织清楚。

### 7. 目标与里程碑管理

这个模块只负责回答“当前工作在服务哪个目标”。

系统的目标治理采用 OKR，项目推进方法采用 PMP，而日常工具承载采用 GitLab。

MVP 阶段至少要支持：

- 为公司设置目标
- 将项目或议题关联到目标
- 用里程碑承接阶段性交付
- 让任务能够向上追溯到公司目标

### 8. 审批、审计与后台治理

这个模块只负责回答“系统如何被管理、监督和追责”。

MVP 阶段至少要支持：

- 对关键动作发起审批
- 记录员工状态变化
- 记录任务分配与升级过程
- 保留操作审计日志
- 在后台统一查看公司、员工、项目与任务情况

## MVP 的最小落地顺序

如果按照实现优先级排序，建议 MVP 按以下顺序落地：

1. 公司与子公司管理
2. 公司自述仓库初始化
3. OpenClaw 员工接入
4. 员工 `SELF` 自述仓库接入
5. 职业化服务建模
6. 单个研发项目的 Issue、里程碑、MR 协作闭环
7. 目标关联
8. 最小后台治理与审计

这样可以先让组织空间成立，再让员工进入组织，再把单项目迭代闭环跑起来，最后补上更强的治理与可视化能力。

## 第一版本的收敛点

这套蓝图要解决的，不是“单个 OpenClaw 能不能做事”，而是下面这些更接近真实组织的问题：

- 一个公司如何快速搭起来
- 多个 OpenClaw 如何进入统一治理
- 不同职业如何保持清晰边界
- GitLab 如何成为稳定的协作底座
- 管理者如何降低搭建复杂度和维护成本

在第一版本中，这些问题都会被收敛到一个更具体的落点上：

- 如何在一个项目中，用 Issue、里程碑、MR 完成一轮又一轮稳定迭代

## 第一批需求列表

以下需求可作为第一版本的需求列表起点，逐步进入后续议题拆解。

### 需求 1：公司级更新播报者

目标是构建一个 `Reporter`，以公司为单位，持续监测公司之下所有项目的更新历史，并按固定周期输出更新播报。

这个需求至少包括以下能力：

- 以公司为范围查看该公司及其子公司下的所有项目
- 按周或按月汇总项目更新
- 识别哪些更新属于“重要更新”
- 说明每项重要更新的主要贡献者是谁
- 生成适合管理层阅读的更新播报

这个需求要解决的问题是：

- 管理者很难快速知道一家公司最近到底完成了哪些重要事情
- 多项目并行时，更新信息容易分散在不同 Issue、里程碑和 MR 中
- 即使知道有更新，也不容易快速识别关键贡献者

从产品形态上看，这个 `Reporter` 更适合作为公司级汇报能力，而不是某个单独项目内部的执行角色。

后续在需求拆解时，可以继续细化为：

- 重要更新的判定标准
- 周报与月报的展示格式
- 贡献者的统计口径
- 汇总范围是否包含子公司项目
