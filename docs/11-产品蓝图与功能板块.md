# 产品蓝图与功能板块

## 产品定位

这套系统是一个建立在 GitLab 与 OpenClaw 之上的统一治理层。

GitLab 是协作底座，但不是全部系统本体。公司、子公司、项目、Issue、里程碑、MR 等核心协作对象，继续尽量沿用 GitLab 语义。

严格来说，即使没有这套产品，仍然可以单独搭建 OpenClaw 服务、手工配置 GitLab 群组、项目、账号、仓库和协作规则，从而完成多智能体协作。但那样的方式有几个明显问题：

- 初始搭建繁杂
- 组织关系需要手工维护
- 权限与职责不易统一
- 公司级治理信息分散
- 后续扩张和长期维护成本较高

因此，这个产品的目标不是替代 GitLab，也不是替代 OpenClaw，而是把原本分散的搭建过程产品化、制度化和可视化。

## MVP 设计原则

MVP 阶段不追求一步到位做完整平台，而是优先做出最小可运行闭环。

当前第一版本，我们先只考虑一个研发项目如何运行，不试图一次完成整套多公司、多项目的完整系统。

这个最小闭环至少要满足四件事：

- 能先创建一个公司并登记员工信息
- 能把 OpenClaw 员工接入到公司治理体系中
- 能通过 GitLab 的项目、Issue、里程碑、MR 完成一个研发项目的基本协作
- 能让管理者看到员工、任务和项目进度之间的对应关系

换句话说，MVP 首先要解决的是“一个项目如何被稳定地迭代”，而不是“穷尽所有组织能力”。

## 功能板块总览

从 MVP 视角出发，系统应优先包含以下功能板块：

- 公司与子公司管理
- 公司自述仓库管理
- OpenClaw 员工接入
- 员工 `SELF` 自述仓库接入
- 职业化服务模型
- 项目与协作流程管理
- 目标与里程碑管理
- 审批、审计与后台治理

## 1. 公司与子公司管理

所有事物都从创建公司开始。

由于公司对应 GitLab 顶级群组，因此系统中的“创建公司”本质上就是创建一个可治理的群组空间。子公司则对应子群组，形式上与公司并无二致，只是挂载位置不同，并继承上级权限与治理语义。

MVP 阶段，这个板块至少要支持：

- 创建公司
- 在公司下创建子公司
- 查看公司树状结构
- 查看群组与子群组的继承关系
- 查看公司对应的目标、预算和员工规模

这个板块回答的问题是：系统里“谁是治理主体”。

## 2. 公司自述仓库管理

既然群组就是公司，那么每一个公司或子公司都不应只是一个空壳群组，而应拥有自己的公司自述仓库。

公司自述仓库的作用，是记录公司级别的长期信息，例如：

- 公司简介
- 公司使命与目标
- 业务边界
- 组织结构说明
- 预算原则
- 协作约定
- 当前重点项目

从 MVP 角度，这个板块至少要支持：

- 检查公司是否已建立自述仓库
- 为新公司创建默认自述仓库
- 为公司自述仓库生成初始化模板
- 将公司自述仓库与公司主页或后台资料页关联

这个板块回答的问题是：这个公司是谁、为什么存在、它遵循什么规则。

## 3. OpenClaw 员工接入

系统当前只允许接入 OpenClaw 作为标准员工来源。

这里需要明确：

- OpenClaw 定义的是员工标准
- GitLab 账号定义的是协作身份
- 模型适配、运行时适配和其他适配器问题由 OpenClaw 自行解决

每位员工都应具备：

- 一个独立的 GitLab 账号
- 一个可被系统识别的 OpenClaw 实体
- 与公司或项目的成员关系
- 可追踪的角色、预算、状态和汇报关系

MVP 阶段，这个板块至少要支持：

- 录入员工的 GitLab 账号
- 建立员工与 OpenClaw 服务的对应关系
- 指定员工所属公司、直属上级和职业类型
- 查看员工当前状态与可承担职责

这个板块回答的问题是：公司里有哪些可用员工，以及他们当前归属于哪里。

## 4. 员工 SELF 自述仓库接入

员工的长期身份信息不放在公司群组下，而放在员工自己的 GitLab 账号下。

`SELF` 先于公司存在，公司负责绑定和授权，而不是替代员工维护身份资料。

每位员工都应拥有一个固定命名为 `SELF` 的自述仓库，用于记录该员工的关键定义，包括：

- `AGENTS`
- `SOUL`
- `SKILL`
- 工作偏好
- 能力边界
- 任务偏好
- 历史沉淀

MVP 阶段，这个板块至少要支持：

- 检查员工是否拥有 `SELF` 仓库
- 将 `SELF` 仓库与员工档案建立关联
- 从 `SELF` 中读取关键定义并展示摘要
- 在分配任务时引用员工 `SELF` 中的能力信息

这个板块回答的问题是：这个员工是谁、擅长什么、适合做什么。

## 5. 职业化服务模型

OpenClaw 的底层能力可以支持多角色，但这只是对 OpenClaw 配置方式的一种期望：系统建议用户不要在同一个服务中配置多种完全不同职责的角色。

MVP 阶段应明确采用职业化建模：

- 一个服务对应一种职业线
- 职业线之下再划分岗位
- 同一岗位下可以有多个分身
- 同一职业内可以定义不同性格和不同处理方式
- 不鼓励同一服务同时承担产品、研发、测试、运维等完全不同职业

例如：

- 产品服务可以包含需求整理、原型分析、方案输出等多个同职业角色
- 研发服务可以包含前端分身、后端分身、接口分身等多个同职业执行单元
- 前端和后端都同属于研发这一职业线
- 但不应让同一个服务既做产品决策，又直接承担研发实现

这样设计有三个好处：

- 权责边界更清楚
- 权限授予更稳定
- 长期维护和替换单个服务更容易

这个板块回答的问题是：员工服务应该如何建模，才能适合长期运维。

## 6. 项目与协作流程管理

这套系统的日常协作仍然建立在 GitLab 机制之上。

MVP 阶段至少应支持：

- 在公司或子公司下创建项目
- 使用 Issue 管理议题、事件、任务
- 使用里程碑组织阶段目标
- 使用 MR 承接变更评审与合并
- 按 GitLab 权限模型配置成员权限

这里的重点不是重新发明一套任务系统，而是先把一个研发项目中的 Issue、里程碑、MR 迭代闭环组织清楚。

这个板块回答的问题是：具体工作如何在项目内流转。

## 7. 目标与里程碑管理

系统的目标治理采用 OKR，项目推进方法采用 PMP，而日常工具承载采用 GitLab。

因此，MVP 阶段至少要支持：

- 为公司设置目标
- 将项目或议题关联到目标
- 用里程碑承接阶段性交付
- 让任务能够向上追溯到公司目标

这个板块回答的问题是：当前工作到底在服务哪个目标。

## 8. 审批、审计与后台治理

即使在 MVP 阶段，也必须保留最基本的治理能力。

至少要支持：

- 对关键动作发起审批
- 记录员工状态变化
- 记录任务分配与升级过程
- 保留操作审计日志
- 在后台统一查看公司、员工、项目与任务情况

这个板块回答的问题是：系统如何被管理、被监督、被追责。

## MVP 的最小落地顺序

如果按照实现优先级排序，建议 MVP 按以下顺序落地：

1. 公司与子公司管理
2. 公司自述仓库初始化
3. OpenClaw 员工接入
4. 员工 `SELF` 自述仓库接入
5. 职业化服务建模
6. 单个研发项目的 Issue、里程碑、MR 协作闭环
7. 目标关联
8. 最小后台治理与审计

这样可以先让组织空间成立，再让员工进入组织，再把单项目迭代闭环跑起来，最后补上更强的治理与可视化能力。

## 这个蓝图解决的核心问题

这套蓝图要解决的，不是“单个 OpenClaw 能不能做事”，而是下面这些更接近真实组织的问题：

- 一个公司如何快速搭起来
- 多个 OpenClaw 如何进入统一治理
- 不同职业如何保持清晰边界
- GitLab 如何成为稳定的协作底座
- 管理者如何降低搭建复杂度和维护成本

在第一版本中，这些问题都会被收敛到一个更具体的落点上：

- 如何在一个项目中，用 Issue、里程碑、MR 完成一轮又一轮稳定迭代

如果这些问题被解决，那么系统就不只是一个代理集合，而是一套真正可经营、可扩张、可维护的多 OpenClaw 协作系统。
