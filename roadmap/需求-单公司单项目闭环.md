# 需求：单公司单项目闭环

## 目标

在第一版本中，建立一条可执行、可审计、可追溯的最短闭环：

- `docs` 仓库负责需求讨论与计划裁决
- 研发总仓负责代码实现与交付
- 两个仓库之间通过统一引用字段建立追溯关系

## 适用范围

- 单公司群组
- 单项目
- 单研发总仓（可包含多个 submodule）

不包含多项目并发调度、跨公司协同和发布运营扩展能力。

## 流程主链

1. 在 `docs` 仓库创建 Issue，输入可为会话、用户故事、用户旅程或会议记录
2. 员工在 Issue 下讨论并收敛
3. 决策者形成“需求计划 MR”
4. 董事会执行强门禁裁决
5. 裁决通过后，计划 MR 合并
6. 项目经理代理人工确认后，在研发总仓创建里程碑、父 Issue 与子任务
7. 研发在总仓执行开发、审查与验收
8. 执行结果回写到计划追溯字段

## `docs` 仓库对象定义

### Issue 必填字段

- `类型`
- `目标项目`
- `当前阶段`
- `决策者`
- `计划版本`

### 计划文档必填字段

- `背景`
- `问题`
- `范围`
- `不做什么`
- `验收标准`
- `拆解建议`

### 计划 MR 必填字段

- `来源 Issue`
- `目标研发仓`
- `计划文档路径`
- `裁决状态`

## 研发总仓对象定义

里程碑、父 Issue、子任务必须统一记录三元引用：

- `docs_issue_id`
- `plan_doc_path`
- `plan_commit_sha`

每个开发 MR 必须回链父 Issue，并继承上述三元引用。

## 状态链

### `docs` 需求状态

`待整理 -> 讨论中 -> 待裁决 -> 已通过 -> 已取消`

### 研发执行状态

`待拆解 -> 待开发 -> 开发中 -> 代码审查中 -> 验收中 -> 已完成 / 已阻塞 / 已取消`

### 门禁规则

- 未通过裁决的计划，不得进入研发执行链
- 已通过裁决的计划，必须经项目经理代理人工确认后才能批量创建研发任务

## 验收标准

1. 任一研发任务都能追溯到唯一 `docs` Issue 与计划版本
2. 任一开发 MR 都能反查 `plan_commit_sha`
3. 未通过裁决的计划无法创建研发里程碑
4. 单项目可完成一次从讨论到验收关闭的全链路演练
