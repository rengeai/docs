# 研发协作体系

## 本文边界

本文只讨论一个研发项目如何运行。

它回答的是：

- Issue 如何分类
- 状态如何流转
- 里程碑如何组织
- MR 如何与 Issue 形成闭环

组织层规则请看 [组织与权限模型](./组织与权限模型.md)。  
代理如何被唤醒和治理，请看 [代理运行与治理机制](./代理运行与治理机制.md)。

## 总体原则

研发协作体系继续沿用 GitLab 的 Issue、里程碑和 MR 模型，但会为 AI 智能体提供更强的结构化分工和可追踪性。

在第一版本中，这篇文档就是系统的主战场。MVP 先聚焦一个研发项目如何通过 Issue、里程碑、MR 完成稳定迭代，而不是一次覆盖全部组织治理场景。

## Issue 体系

Issue 是一切工作的起点，也是需求、事实、任务和讨论的统一容器。

### 三种 Issue 类型

在这套体系里，沿用三类核心语义：

- 议题：表示需求，是希望被实现的目标或能力
- 事件：表示已经发生的事实，例如线上故障、事故、异常告警
- 任务：表示基于议题或事件拆解出来的执行单元

这三类对象共享同一个 Issue 容器，但语义和流转目的必须区分。

### 三类 Issue 的关系

- 议题回答“为什么做”
- 事件回答“发生了什么”
- 任务回答“具体怎么做”
- 议题通常会拆分出多个任务
- 事件可以直接触发任务
- 任务应尽量挂靠在某个议题或事件之下

### 建议的 Issue 字段

- 类型
- 标题
- 描述
- 状态
- 优先级
- 受让人
- 父级 Issue
- 关联项目
- 里程碑
- 可选目标关联

## 任务追踪链路

任务层级必须可追溯到公司目标。建议采用如下追踪链：

1. 公司目标
2. OKR
3. 项目
4. 议题或父级 Issue
5. 子 Issue
6. 实际执行任务

这意味着任何一个底层任务都不应孤立存在，它必须能解释自己服务于哪个更高层级目标。

## 状态流转

Issue 的状态管理遵循 GitLab Issue 的管理理念，并在此基础上做适配。

建议采用以下标准状态流转：

`待整理 -> 待执行 -> 进行中 -> 评审中 -> 已完成`

同时保留一个侧向状态：

`已阻塞`

终态包括：

- 已完成
- 已取消

### 状态含义

- `待整理`：尚未进入明确排期
- `待执行`：已准备好执行，但尚未开工
- `进行中`：已有代理开始处理
- `评审中`：工作已提交，等待审阅或审批
- `已阻塞`：因依赖、权限、资源或决策问题被阻塞
- `已完成`：工作已完成并验收
- `已取消`：工作被明确终止

### 状态设计原则

- Issue 是责任归属、讨论过程和状态追踪的主入口
- 状态变化应尽量简洁，避免过度细分
- 里程碑用于聚合相关 Issue，而不是替代 Issue 本身
- 父子 Issue 用于表达任务拆解和目标追踪关系

## 原子认领机制

任务从 `待执行` 进入 `进行中` 必须经过原子认领。

- 一次只能有一个代理拥有一个任务
- 如果两个代理同时尝试认领同一个任务，其中一个必须失败
- 失败应返回标准 `409 Conflict`

这条规则用于保证任务所有权清晰，避免重复劳动和状态污染。

## 里程碑与路线图

### 路线图

路线图用于承接中长期规划，是里程碑与 Issue 集合在时间维度上的表达方式。

路线图需要回答的问题包括：

- 当前阶段最重要的建设方向是什么
- 每个方向由哪些项目承接
- 每个项目由哪些议题组成
- 每个阶段对应哪个里程碑
- 当前进度如何
- 风险点在哪里

建议的路线图层次：

- 战略路线图：对应组织级目标
- 项目路线图：对应单个项目的季度或月度安排
- 执行路线图：对应当前迭代内的重点工作

### 里程碑

里程碑沿用 GitLab 的理念，用来承载一个阶段内要达成的一组结果，而不是单一任务。

里程碑的作用：

- 组织同一阶段内的多个 Issue
- 为路线图提供时间切片
- 统一衡量阶段性交付进度
- 帮助管理员识别延期和阻塞风险

里程碑的使用原则：

- 一个 Issue 可以归属于一个明确里程碑
- 一个里程碑下可以聚合多个相关 Issue
- 里程碑应服务于项目目标或 KR，而不是独立存在
- 里程碑完成不等于目标完成，但应代表阶段性成果交付

## MR 体系

MR 是变更进入主线的治理入口，也是人与 AI 协作质量控制的关键环节。

### MR 的目标

- 解释为什么要修改
- 描述修改了什么
- 展示如何验证
- 暴露潜在风险
- 沉淀评审过程

### AI 场景下的 MR 特点

- MR 可以由 AI 智能体发起
- MR 必须附带结构化变更说明
- 关键 MR 需要人类或高权限智能体审批
- 审批意见和测试结果要形成可追踪记录
- MR 必须能关联回对应 Issue、里程碑与上层目标

### 建议的 MR 模板字段

- 背景
- 变更范围
- 验证方式
- 风险说明
- 回滚方案
- 关联 Issue
- 关联里程碑
- 关联目标

## 从 Issue 到 MR 的闭环

建议形成如下闭环：

1. 从目标、路线图或里程碑产生 Issue
2. 由 AI 智能体原子认领任务
3. 在项目内推进设计与实现
4. 通过 MR 进入评审流程
5. 合并后更新里程碑、路线图和目标状态
6. 在复盘中沉淀经验与制度优化

这个闭环是第一版本最重要的运行基础。
