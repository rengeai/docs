# 单公司单项目闭环

## 背景

第一版本聚焦单公司群组下的单项目协作，不展开多项目并发和跨公司协同。

## 问题定义

当前需要建立从 `docs` 需求讨论到研发总仓交付的稳定主链，避免需求与代码脱节。

## 范围与非范围

- 范围：`docs` 提案、董事会裁决、人工确认、研发执行、追溯回链。
- 非范围：多公司调度、发布运营扩展、跨组织权限同步。

## 方案设计

- `docs` 仓库承载 Issue 讨论和计划 MR。
- 董事会以强门禁裁决计划是否可执行。
- 项目经理代理人工确认后创建研发里程碑、父 Issue、子任务。
- 研发总仓执行开发、审查、验收。

## 执行计划

1. 在 `docs` 发起 Issue 并讨论。
2. 形成计划 MR 并进入裁决。
3. 裁决通过后创建研发执行对象。
4. 执行 MR 交付并回填状态。

## 风险与回滚

- 风险：字段不统一导致追溯断裂。
- 回滚：撤销未通过裁决的执行对象并回退到 `Review`。

## 验收标准

- 任一研发任务可追溯到唯一 `docs` Issue 与计划文件。
- 任一开发 MR 可反查 `plan_commit_sha`。
- 未 `Approved` 的提案无法进入研发执行链。
